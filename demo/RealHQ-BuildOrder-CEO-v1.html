1 Honest Assessment — What Went Wrong and Why

This is not a blame document. It is a clarity document.

Both sides contributed to the gap between what was envisioned and what was built. Understanding exactly how that happened is the only way to prevent it happening again. Read this section before reading anything else.



1.1 What RealHQ was always supposed to be

RealHQ is an execution engine for commercial property owners. Not a dashboard. Not a data display. An engine that does the work that currently requires five phone calls, three intermediaries, and five days of waiting — automatically, in seconds, via software.

The defining test for every feature: does the owner have to contact anyone, wait for anyone, or do anything manual after clicking a button? If yes — it is not finished.

The product we are building

The product that was built

Owner uploads insurance schedule. Binding quotes from AIG, Chubb, Lloyd's appear within 60 seconds. Owner clicks confirm. Policy placed. Commission captured.

Owner fills in a form. Ian reviews it. Ian contacts brokers. Quotes arrive by email days later labelled "Illustrative Market Rates".

Energy tariff extracted from uploaded bill. Live supplier APIs return best rate. Owner confirms. Contract switched by software.

Energy comparison labelled "Illustrative Tariff Benchmarks". No live data. No switch execution.

Lease uploaded. ERV calculated from live CoStar data. Leverage score generated by AI. Renewal letter drafted. One click to send.

Rent benchmark shown as a static percentage. No ERV source. No letter. No action.

Property address entered. Satellite image, boundary, planning history, EPC, flood zone, comparables all load automatically. No user action required.

Manual setup wizard asking the owner to type in their own data.

Every CTA triggers a real API call, document generation, or executed workflow.

A "track your submitted request" page — meaning a human at RealHQ responds to each request manually.



1.2 Where both sides went wrong

  • Our side: We showed a working prototype as a reference and assumed the builder would extract the product philosophy — not just the visual structure. We said "build something like this demo" without specifying "every number must come from a live API, every button must execute a real action, no human at RealHQ should touch anything after the owner clicks."

  • Our side: The original brief was strong on vision but left the execution model open to interpretation. "Propra places direct with carriers" was written — but what that means technically (API call, not phone call) was never spelled out.

  • Build side: The brief and demo were read as a UI design brief. Data was displayed. The product logic was built around a human intermediary reviewing form submissions. The core principle — software executes, software does not recommend — was not understood or built.

  • Build side: When asked about features like insurance and energy, the response was to label them "illustrative" rather than flag that the API integrations hadn't been built yet. That is not a working feature — it is a placeholder presented as a feature.



1.3 What the CEO built responses actually show

Looking at the specific build response (commit 1698784d) that triggered the CEO's frustration:

  • "Service delivery tracking" — a page where Ian can update the status of requests. This is a CRM for a manual service business, not a software product.

  • "Self-service portfolio setup wizard" — asking users to type in their own data. RealHQ should extract data from uploaded documents automatically. There should be no wizard.

  • "Fixed misleading demo states" — removing hardcoded "done: true" from Insurance and Energy steps. These were placeholders being presented as completed features.

  • "Labelled benchmarks clearly" — renamed to "Illustrative Market Rates". This is correct as a temporary fix but is not a finished feature. Illustrative means the API isn't wired yet.

None of this is wrong work in isolation. The problem is that this work was done instead of wiring real APIs — and was presented as completing features that are still hollow.



2 What Was Missed From the Original Demo

Four elements not yet in the spec or prototype that must be added

These were visible in the demo all along

After a thorough re-review of the original demo artifact, four components were identified that are not yet in the RealHQ spec or prototype. Each of them is significant. They are added to the spec and prototype as part of this update.



2.1 NOI Optimisation Bridge — the most important panel on the dashboard

The demo shows a panel that stacks every open opportunity as an incremental contribution to monthly NOI, building from current to projected:

  • Current Monthly NOI: $487,300

  • + Rent uplift (6 tenants): +$3,650

  • + Insurance rearrangement: +$2,842

  • + Refinance (2 properties): +$4,083

  • + Utility switching & solar: +$1,842

  • + CAM recovery: +$1,217

  • + Wynwood flex conversion: +$1,550

  • Projected Monthly NOI: $502,484

  • Implied portfolio value uplift at 6.5% cap rate: +$2.77M

This is the single most compelling element in the entire demo. It shows the owner in one glance what their portfolio is worth today and what it will be worth when all actions are completed. Every open opportunity contributes a bar. As actions are completed, bars turn solid green. The +$2.77M figure at the bottom is what makes an owner act.

This must be added to the dashboard. It is not optional.



2.2 Refinance Centre table inline on the dashboard

The demo shows an inline table on the dashboard (not just a summary card) with columns: Property / Current Rate / Market Rate / Balance / DSCR / Annual Saving / Action. Each row has a status button inline: "Refi now" (green), "Explore" (blue), "Monitor" (grey), "Hold" (amber). This is far more useful than a summary card — it is a decision table that tells the owner exactly which loans to act on and why. Add to dashboard.



2.3 Acquisition Pipeline inline on the dashboard

Two property deal cards appear directly on the dashboard — one off-market, one on-market — each with AI fit rating (High fit / Med fit), asking price, cap rate, current NOI, occupancy, and a short AI analysis summary. Subtitle: "AI screens 400+ FL listings daily against your criteria." This surfaces the acquisitions scout directly into the dashboard rather than burying it in its own screen. Add to dashboard.



2.4 Market Benchmarking panel

A right-column panel comparing the portfolio against the FL commercial median for the current quarter. Metrics: Cap Rate / NOI Margin / Occupancy / Rent per sqft / OpEx per sqft / DSCR / Insurance per sqft. Each has a progress bar showing portfolio vs market, with status labels (Above mkt / Below mkt / Overspending / Strong). Red labels are immediate calls to action. Below the table is a 6-month NOI Trend mini line chart. This is more powerful than the current geographic spread + asset mix row because it directly shows where the portfolio is underperforming relative to market. Add to dashboard.



3 Feature Build Order

Fastest path to revenue, all features visible, never stop at one

The governing principle for this build order

Three criteria determine sequence: (1) speed to earn commission — which features generate RealHQ revenue fastest once live; (2) data complexity — simpler data inputs mean faster time to live; (3) all features are visible in the UI from day one, progressively activating as APIs are wired. The dashboard prototype shows every feature. Nothing is hidden. Features light up as they go live. We stop building when all features are working and tested — not before.



3.1 Dashboard-first principle

Before any feature is built, the full dashboard prototype is live in production. Every opportunity category shows on screen — insurance, energy, solar, 5G, planning gain, refinance, rent uplift, CAM recovery, value add, acquisitions. If the underlying API is not yet wired, the card shows a "Coming soon" or "In progress" state with the expected saving clearly shown. The owner sees the full platform immediately. Data populates progressively as APIs go live.

For new users who have not yet uploaded documents: a seeded demo portfolio is shown by default. The moment they upload their first real document, their real data replaces the demo data for that section. The dashboard is never empty.

3.2 What "never stop after one feature" means in practice

  • All waves run in parallel once started. Wave 2 begins the moment Wave 1 insurance is live — not after Wave 1 is "complete". Wave 3 begins when Wave 2 energy is live. The platform is always moving on multiple fronts.

  • Every feature in every wave has a named owner and a named API integration. "In progress" states are acceptable. "Illustrative" states are never acceptable in production.

  • The dashboard always shows all features. A feature with no live data shows its expected value in a "pending" state. This keeps the vision visible to every user and creates demand to activate the pending features.

  • Testing is part of the feature — not a separate phase. A feature is not done until it has been tested end-to-end with a real document from a real client. Not demo data. Real data.

  • Commission tracking is built from Wave 1. Every action that earns RealHQ commission is recorded in the platform the moment it is executed. This is both a product feature (clients can see what RealHQ has earned them vs charged them) and a business intelligence tool.

7 CEO Letter — What to Send and What It Should Say

The following is the recommended message to send to the CEO alongside this document and the other attachments. It is direct, takes responsibility on both sides, explains the change clearly, and gives concrete next steps.

Subject: RealHQ — New direction, honest review, and exactly what we are doing about it



I want to be direct with you about where we are, what went wrong, and the concrete steps we have taken to fix it. No blame — just clarity.

WHAT REALHQ WAS ALWAYS SUPPOSED TO BE

RealHQ is an execution engine. A property owner uploads an insurance schedule, binding quotes from AIG, Chubb, and Lloyd's appear within 60 seconds, they click confirm, the policy is placed. No calls. No emails. No waiting. Same for energy switching, refinancing, planning applications, procurement, and every other feature. The software does the work — it does not recommend that someone else does the work.

What was shipped is a data display product built around a manual service business workflow. The key signals: a setup wizard asking users to type in their own data; a page for tracking whether a human has responded to a quote request; insurance comparisons labelled "Illustrative Market Rates" rather than live carrier quotes; energy data labelled "Illustrative Tariff Benchmarks". These are placeholders presented as features.

Our side: we showed a demo and assumed the builder would extract the product philosophy from it. We did not explicitly specify that every button must trigger a real API call, that no human at RealHQ should touch anything after an owner clicks, that "illustrative" is never acceptable in a shipped feature.

Build side: the brief and demo were read as a UI design brief. The execution model — software places the insurance, software switches the energy, software submits the planning application — was not understood and not built.

WHAT WE HAVE DONE DIFFERENTLY THIS TIME

Four documents and one prototype are attached. Together they leave nothing open to interpretation:

1. Master Spec v3.0 — all 28 screens, every API integration named, every automation trigger specified, acceptance criteria for every feature.
